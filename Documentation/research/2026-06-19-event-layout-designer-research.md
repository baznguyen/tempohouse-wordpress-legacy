# Event Layout Designer — Competitive Research & Gap Analysis
**Date:** 2026-06-19  
**Scope:** Event floor planning platforms reviewed for feature patterns and gaps, framed against the TEMPO House Event Layout Designer MVP spec  
**Platforms reviewed:** Social Tables / Prismm, AllSeated, Planning Pod, Merri (now Tripleseat), Cvent Event Diagramming, EventDraw, RoomSketcher, SmartDraw, Planning Pod, zkipster / PlanSeats (seating assignment layer)

---

## 1. Industry Workflow Reality — Three Distinct Use Cases

Event floor planning tools conflate three workflows that serve different stakeholders. Understanding this prevents over-engineering the MVP.

### 1A — Venue-to-Planner Workflow (Social Tables, Merri)
Venues invite event planners into shared workspaces. Planners need large asset libraries and real-time co-editing. This is a B2B SaaS feature set — overkill for a single venue running its own events. Not required for TEMPO MVP.

### 1B — Planner-to-Client Workflow (AllSeated, Prismm)
Staff designs layout → share read-only or comment-enabled link → client views and approves. This is exactly TEMPO House's use case. Client approval timestamp and version history are the operational value, not 3D walkthroughs.

### 1C — Venue-to-Guest Workflow (PlanSeats, zkipster, RSVPify)
Guest names assigned to specific seats, dietary tags visible per table, catering counts per zone. This is Phase 2+ for TEMPO — high ROI when events exceed 40 people but not required for an initial layout tool.

**Takeaway:** The TEMPO Event Layout Designer is primarily 1B (Planner-to-Client) with elements of 1C on the roadmap.

## 2. Scale Accuracy — The DPI Problem

### How Industry Tools Handle This

- **EventDraw / Cvent**: Accept DWG/CAD imports with embedded scale metadata. All objects inherit millimetre accuracy from the source drawing.
- **RoomSketcher**: Captures room dimensions at project creation (length × width); stores all object dimensions in real-world units; renders at current viewport DPI; exports at 300 DPI (vector or high-DPI raster).
- **Web-native tools**: Pixel-based canvas with a conversion factor stored per-layout (e.g., `scale: 4px/mm`).

### The Gotcha
At 96 DPI (web default), 1px ≈ 0.2646mm. A canvas snapshot PDF at 96 DPI looks blurry at print. At 300 DPI (standard print quality), objects scale by 3.125×. If scale metadata is wrong, a 3m table prints at 96cm.

### For TEMPO House
- Store all furniture dimensions in mm (source of truth).
- Render on canvas at `1mm = SCALE_PX` where `SCALE_PX` is set per-layout (default: `4px/mm`).
- On PDF export, render canvas at `pixelRatio: 3` (Konva's native option) and embed scale label on the document (e.g., "1:50 — 1cm = 500mm").
- Test: measure the printed bar counter against the architect PDF dimensions. They must match.

## 3. PDF Generation — Approach Comparison

| Approach | Quality | Complexity | Notes |
|---|---|---|---|
| Browser print (`window.print()`) | Poor — layout breaks | None | Do not use — Bailey explicitly rejected this |
| Canvas snapshot (Konva `toDataURL`) | Medium — 96 DPI raster | Low | Fast. Blurry at print unless pixelRatio: 3 |
| Canvas snapshot + jsPDF | Good — high-DPI raster | Low–Medium | Best for MVP. Mix image + text layers in jsPDF |
| Headless browser (Puppeteer server-side) | Excellent — 300 DPI vector | High | Not available in WP plugin without server process |
| SVG export → PDF | Excellent — infinite scale | Medium | Requires Konva-to-SVG conversion step |

**Recommended approach for TEMPO MVP:** Konva `stage.toDataURL({ pixelRatio: 3 })` to get a 3× canvas image, then jsPDF to compose:
1. Cover block: event name, date, time, venue name, scale label
2. Floor plan image (full page, centred with margins)
3. Zone legend (coloured chips + zone names)
4. Notation appendix table (notation number → note text)
5. Footer: "Generated by TEMPO House Event Layout System — [date]"

This produces a "designed" PDF, not a print-out, because jsPDF controls typography, margins, and layout independently of the canvas.

## 4. Access Control Models (Client Sharing)

| Model | Supported by | Required for TEMPO |
|---|---|---|
| View-only link (anyone with link) | AllSeated, Planning Pod | ✓ MVP |
| View + inline comment | Prismm, EventDraw | ✓ Phase 2 |
| Edit link (co-design) | Social Tables | Phase 3 / optional |
| Approval workflow (approve / request changes buttons) | Prismm, Releventful | Phase 2 |
| Version history (V1, V2, Vn) | Planning Pod, Prismm | Phase 2 |

**For MVP:** A shareable read-only URL containing a layout token is sufficient. The client cannot edit but can view the layout and its event details.

**Phase 2 minimum viable sharing:** Add an "Approve layout" button on the client view. One click generates a timestamped confirmation record visible in the staff dashboard. This is the equivalent of an email confirmation but anchored to the exact layout version.

## 5. Annotation/Notation — Industry Split

Platforms treat annotations in two distinct ways:

**Internal markup (staff-only notes):** Notes like "Client said no bar table in south corner" — hidden from client-facing views. Used in Planning Pod and Social Tables.

**Communication markup (client-visible):** "Entrance here," "Welcome table," "Photo booth" — styled to match the design aesthetic. These appear in the shared view and in the PDF. Used in Prismm and EventDraw.

**Technical markup:** Capacity numbers, flow arrows, fire exit paths (compliance). Often a separate locked layer. Used in Cvent, RoomSketcher.

**For TEMPO House:** Annotations should be client-visible by default (they describe layout intent, not internal logistics). Keep a private "staff note" flag for annotations that should be internal only (hidden from client share link, still visible in PDF appendix with "(Staff only)" tag).

## 6. Feature Gaps Identified — Not in Bailey's Spec

These features exist in industry tools and have genuine value for TEMPO House. Scored by **ROI** (operational value) vs **effort** (build complexity).

| Feature | Where used | ROI | Effort | Recommendation |
|---|---|---|---|---|
| **Capacity estimator** (sqft/person live calculation) | EventDraw, RoomSketcher | High | Low | Include in MVP — 1 formula, renders in props panel |
| **Layout presets** (cocktail, seated, theatre, etc.) | Planning Pod, Merri | High | Low | Include in MVP — JSON preset arrays |
| **Constraint markers** (fire exits, pillars — locked, non-draggable) | Cvent, RoomSketcher | High | Low | Include in MVP — uses structural overlay layer |
| **Client approval timestamp** (click to approve, timestamped record) | Prismm, Releventful | High | Low | Phase 2 |
| **Version snapshots** (auto-save on major change; visual diff) | Planning Pod, Prismm | Medium | Medium | Phase 2 |
| **Inline spatial comments** (client comments anchored to object) | Prismm, EventDraw | Medium | Medium | Phase 2 |
| **Guest seating assignment** (guest list → table seat) | Prismm, PlanSeats, zkipster | High | High | Phase 3 |
| **Notes-to-catering export** (extract annotations as catering brief) | EventDraw (partial) | Medium | Low | Phase 3 |
| **Asset library by event type** (load preset + client customises) | Planning Pod, Merri | Medium | Low | Phase 2 |
| **3D walkthrough / preview** | AllSeated, Prismm, Tripleseat | High marketing | Very High | Roadmap only — 3D is overkill for boutique venue |

## 7. Furniture Taxonomy — Industry Standard vs TEMPO House

Standard event planning platforms organise their palette into 5 tiers. Mapping to TEMPO House context:

| Tier | Industry items | TEMPO House equivalent |
|---|---|---|
| Seating | Chairs (banquet, chiavari, lounge), high stools | Theatre chair, bar stool, lounge, single chair |
| Tables | Round 60"/72", rect, cocktail, console | All specified variants by mm |
| Staging / AV | Stage/riser, podium, DJ booth, screen, projector | Media wall backdrop, projector |
| Décor | Draping, centrepieces, uplighting | Flower centrepiece (expandable later) |
| Catering / Flow | Buffet stations, bar, beverage carts | Future: "drinks station," "food station" |

**Missing from Bailey's spec that are standard:**
- **Flow arrows / directional markers** (indicate traffic flow, entrance/exit direction)
- **Text label elements** (heading labels for zones, not just notation pins)
- **Scale ruler display** on canvas (shows printed scale, e.g. "1cm = 50cm")

## 8. Key Insights for TEMPO House Build

1. **Speed is the #1 staff UX requirement.** EventDraw's "90-second layout" claim is the benchmark. Staff should go from blank canvas to a shareable cocktail reception layout in under 3 minutes using a preset + quick customisation. Presets and a fast toolbar with keyboard shortcuts are how you achieve this.

2. **Client approval is the most undervalued feature.** Once a layout is shared, the current industry standard is "email me." Prismm's one-click approval button (timestamped, layout-version-locked) eliminates email back-and-forth and creates a clear audit trail. High value, low effort.

3. **Scale accuracy matters more than it seems.** Clients who receive a PDF and then attend the actual event notice immediately if the layout doesn't match reality. Capturing room dimensions at layout creation and baking a scale label into every PDF is table-stakes for trust.

4. **PDF quality is a proxy for venue quality.** A blurry, browser-print layout PDF signals a casual operation. A clean, designed PDF with the venue brand, scale label, zone legend, and notation appendix signals a professional event partner. This is where TEMPO's brand investment pays off.

5. **Guest seating is the natural evolution.** Once layout management is working well and events are filling up, the question becomes "who sits where?" The connection between a layout table and a guest name/dietary flag is the bridge between the floor plan tool and the CRM. Plan the DB schema to support this from the start, even if the UI comes later.
